# XiAgent UI 审查

## Overview

这是 XiAgent 项目级 UI 页面审查技能。审查目标是确认指定 UI 版本目录内的页面已经完成真实数据对接、没有测试静态数据残留、内容和设计文档一致、代码遵守项目边界，界面布局在可运行页面中没有遮挡、错位或明显不美观的问题，并且最终能通过网页按真实用户交互完成全流程测试。

## Scope Gate

- UI 代码必须位于 `ui/<version>/...`，例如 `ui/V1/...`；`ui/` 只是 UI 根目录，不是 UI 代码目录。
- 如果用户没有指定版本，先检查 `ui/*` 和当前 git diff。只有一个明确候选时直接审查；存在多个候选且无法从变更判断时，先问用户要审查哪个版本。
- 不要把非 UI 代码搬进 `ui/` 根目录。需要共享 UI 能力时，也放在明确版本目录或经用户确认的 `ui/shared/...` 结构中，并说明跨版本影响。
- 审查前先读取项目约束：`AGENTS.md`、相关 `docs/design/`、`docs/project-architecture/`、`docs/development/` 文档，以及目标 UI 版本的入口、路由、数据访问层和测试。

## Review Workflow

1. 确定审查对象：目标路径、页面清单、关联功能、设计文档、已变更文件和启动方式。
2. 建立页面到数据源的映射：每个页面展示的数据来自哪个 API、服务、状态管理或运行时输入。
3. 搜索测试数据残留：检查 `mock`、`fixture`、`dummy`、`fake`、`sample`、`static`、`hardcoded`、`测试`、`示例`、`假数据`、`TODO`、`FIXME` 等关键词。静态文案、空状态文案和字段标签允许存在；业务列表、统计值、详情内容、用户/项目/任务/资产数据不能用静态数组冒充真实数据。
4. 对照设计文档：页面结构、字段、交互、状态、权限、术语、错误提示、空状态、加载状态必须与设计文档一致。设计文档缺失或过期时，不要自行补脑通过；把它列为问题或风险。
5. 检查项目边界：UI 不得直接访问 SQLite、资产文件路径、后端内部类、模块内部实现或第三方适配细节；必须通过稳定 API、服务接口或 UI 版本内的数据适配层访问数据。
6. 检查代码规范：沿用目标 UI 版本现有框架、目录、状态管理、类型、错误处理和测试模式。避免把临时调试、控制台噪声、演示入口、硬编码账号或本地 URL 留在生产路径。
7. 检查界面质量：能运行页面时启动本地应用，优先使用 Codex 内部浏览器检查桌面和移动宽度。重点看遮挡、溢出、文本截断、层级覆盖、滚动异常、按钮/表单状态、空/错/加载态、密集区域排版、组件间距和视觉一致性。布局和交互判断可调用 `$ui-ux-designer` 作为参考。
8. 运行基础验证：优先使用目标 UI 版本已有的 lint、typecheck、test、build 命令。没有现成命令时，说明无法验证的缺口，不要声称已通过。外部浏览器自动化 CLI、旧 e2e 脚本、截图测试或组件孤立渲染只能作为辅助回归证据，不能替代第 10 步的真实交互验收。
9. 检查任务快照语义：任务详情必须按该任务持有的 workflow snapshot 和节点 UI 配置展示。历史任务仍显示旧控件是正确行为；不得通过修改旧 `control_id`、variant、mode 或 binding 的前端实现来让历史 snapshot 看起来像新配置。
10. 运行真实用户网页全流程测试：启动后端和 UI，必须优先使用 Codex 内部浏览器（in-app browser / Browser plugin）按真实用户操作路径完成至少一个端到端主流程。流程必须使用真实服务/API、真实页面点击、输入、等待和可识别业务输入，覆盖登录或进入项目、打开目标页面、执行核心操作、确认数据保存/刷新/跳转/结果展示。不能用外部浏览器自动化 CLI、旧 e2e 脚本、组件孤立渲染、直接调用接口、数据库或 localStorage 预置、跳过权限、只看截图替代。若 Codex 内部浏览器不可用，必须列出阻塞原因和待人工验证步骤，不能声称真实交互测试已通过。若验收工作流 UI 配置变更，必须先确认后端已重新加载工作流目录，再新建任务验证新 snapshot；只有在显式迁移 snapshot/config 后，历史任务才可作为新样式验收对象。

常用搜索命令示例：

```powershell
git status --short
rg --files "ui/<version>"
rg -n --glob "!node_modules/**" --glob "!dist/**" "(mock|fixture|dummy|fake|sample|static|hardcoded|测试|示例|假数据)" "ui/<version>"
rg -n --glob "!node_modules/**" --glob "!dist/**" "(localhost|127\.0\.0\.1|example\.com|TODO|FIXME)" "ui/<version>"
```

## Issue Report

审查完成后先输出问题清单，不要直接修改，除非用户已经明确要求边审边改。问题按严重程度排序：

| Level | Meaning |
| --- | --- |
| P0 | 页面无法运行、核心流程不可用、错误数据来源、权限或项目边界严重违规 |
| P1 | 真实数据未接通、测试静态数据残留、设计文档关键内容不匹配 |
| P2 | 代码规范、状态处理、测试覆盖、响应式布局或可维护性问题 |
| P3 | 文案、间距、视觉细节、轻微一致性问题 |

每个问题写清：

- 文件和行号，或页面路径和可复现步骤。
- 现象和证据。
- 违反的设计文档、项目约束或 UI 质量标准。
- 建议修复方向。
- 建议归属：文档校验、数据对接、代码规范、视觉布局、测试验证。

报告末尾必须列出已执行的验证命令、真实用户网页全流程测试路径、关键输入和结果、未能执行的验证及原因。然后询问用户选择：

```text
发现以上问题。你希望我用哪种方式处理？
1. 智能体修改：我负责拆分任务，多个智能体分别处理文档/代码校验、代码审查、修改、测试和复审，并循环到审查无问题。
2. 当前任务直接修改：我在当前会话内按优先级逐项修复并验证。
3. 只保留审查报告：暂不修改。
```

## Agent Fix Mode

用户选择“智能体修改”时，当前 Codex 作为协调者，不直接把所有问题混在一起改。先使用可用的多智能体或 subagent 工具；如果当前环境没有真实 subagent 能力，必须告知用户并改为当前任务内分阶段执行，不能假装已经并行派发。

拆分任务时默认使用这些角色：

| Role | Responsibility |
| --- | --- |
| 文档与契约校验智能体 | 对照 `docs/design/`、架构约束和页面需求，确认问题是否真实、文档是否缺失或过期 |
| 真实数据对接智能体 | 移除测试静态数据，接入真实 API/服务/状态管理，并补齐加载、错误、空状态 |
| UI 代码规范智能体 | 检查目录边界、类型、状态管理、错误处理、可维护性和现有模式一致性 |
| 视觉布局智能体 | 参考 `$ui-ux-designer`，检查排版、遮挡、响应式、组件状态和视觉一致性 |
| 测试验证智能体 | 运行 lint、typecheck、test、build，并用 Codex 内部浏览器按真实用户交互完成全流程测试，记录证据 |
| 复审智能体 | 在修改后重新执行本 skill 的审查，确认问题是否关闭且未引入新问题 |

每个智能体任务包必须包含：目标路径、相关问题、禁止越界修改的文件范围、需要读取的设计文档、验收标准、必须运行或建议运行的验证命令、输出格式。多个智能体可能修改同一文件时，由协调者串行化这些任务，避免互相覆盖。

修复循环：

1. 协调者把问题拆成独立任务，先处理 P0/P1，再处理 P2/P3。
2. 智能体完成修改后，协调者审查 diff，确认没有绕过真实数据、没有新增静态测试数据、没有破坏 UI 目录边界。
3. 测试验证智能体运行验证并给出原始命令结果摘要，同时记录 Codex 内部浏览器真实用户全流程测试的访问 URL、操作路径、输入、页面结果、刷新/跳转后的持久化确认和截图或日志证据。
4. 复审智能体重新按本 skill 审查目标 UI 版本。
5. 如果仍有 P0/P1/P2，回到第 1 步继续拆分和修复。P3 需要修复或明确让用户接受后才能收尾。

## Direct Fix Mode

用户选择“当前任务直接修改”时，在当前会话内按同一审查问题优先级执行。修改前先说明将改哪些文件和原因；涉及行为变化时优先补测试或更新现有测试；修改后必须重新运行可用验证，使用 Codex 内部浏览器按真实用户交互完成全流程测试，并再次执行本 skill 的复审步骤。

## Common Mistakes

- 只看页面截图不查数据来源：必须建立页面字段到真实数据源的映射。
- 把 mock 文件改名成 service 就算接入：必须确认运行时调用真实 API、服务或状态管理。
- 把 UI 代码直接放在 `ui/` 根目录：必须落在 `ui/<version>/...`。
- 设计文档不一致时自行选择一边通过：必须列出差异，让用户或文档修复流程确认。
- 只跑 build 不看页面：UI 审查必须覆盖实际排版、遮挡、响应式和状态展示。
- 只跑接口测试、组件测试、外部浏览器自动化 CLI 或截图测试就验收：最终必须用 Codex 内部浏览器按真实用户交互走完整主流程。
- 用历史任务验收新工作流 UI 配置，或把旧控件实现改成新控件效果来掩盖旧 snapshot；正确验收是新建任务生成新 snapshot，或先显式迁移目标历史 snapshot/config。
- 智能体模式下没有复审循环：修改、测试后必须重新审查，直到没有未解决问题。
