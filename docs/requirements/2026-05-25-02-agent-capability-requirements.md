# Agent 能力需求文档

## 背景

XiAgent 当前以开发者维护的 YAML/JSON 工作流契约为核心，适合承载确定性的 DAG 流程、节点输入输出、条件分支、执行记录和测试验证。

这类固定工作流可以稳定处理明确流程，例如资产整理、角色抽取、变体匹配、图像提示词生成和分镜生成。但在实际使用中，用户目标和用户反馈经常具有开放性，例如：

```text
帮我整理这个剧本里的角色资产。
这个提示词不够像原作。
服装更华丽，但不要改变发色。
这张图的风格不对。
这个角色识别错了。
```

这些场景需要系统具备一定的自然语言理解、判断、修订、评审和协调能力。Agent 能力应在不破坏工作流可控性、可测试性和可追踪性的前提下，为工作流节点和工作流执行过程提供智能辅助。

## 问题陈述

当前固定工作流模式存在以下不足：

- 用户无法方便地对节点中间结果提出自然语言修改要求。
- 节点输出不满意时，通常只能整体重跑或人工调整输入。
- 如果为每一种修订场景预定义独立工作流，会造成模板数量膨胀。
- 工作流本身不适合表达开放式判断、质量评审和灵活修订。
- 用户反馈可能只影响当前节点，也可能影响上游节点，系统需要具备判断能力。
- 节点输出质量目前主要依赖单次模型生成，缺少统一评审与自修机制。
- 开放式用户目标需要一种方式选择、串联或驱动已有工作流。

因此，XiAgent 需要引入受控 Agent 能力，用于补足固定工作流在开放式判断和交互式修订上的不足。

## 总体目标

Agent 能力的总体目标是：

```text
在保持工作流工程可控的基础上，引入受控的智能判断、节点处理、质量评审和用户反馈修订能力。
```

具体目标包括：

- 支持在工作流节点内部完成复杂智能处理。
- 支持对节点输出进行质量、约束和一致性检查。
- 支持用户对关键节点结果提出自然语言反馈。
- 支持根据反馈生成修订建议或修订结果。
- 支持保留原始结果和修订版本，避免直接覆盖历史记录。
- 支持识别反馈是否可能影响上游节点。
- 支持轻量级工作流选择和跨工作流协调。
- 保持资产、权限、持久化和工作流执行边界清晰。
- 理解用户的开放式目标，选择或串联已有工作流，并在工作流之间传递结构化输入。

## 非目标

Agent 能力不追求以下目标：

- 不支持通用工作流循环。
- 不支持 Agent 动态生成并立即执行新的工作流模板。
- 不支持 Agent 绕过工作流运行器直接执行节点。
- 不支持 Agent 直接访问 SQLite 或内部数据库表。
- 不支持 Agent 直接拼接或读写资产文件路径。
- 不支持 Agent 直接覆盖历史节点输出。
- 不支持 Agent 绕过 `AssetService`、用户服务或权限检查。
- 不要求第一版自动从任意上游节点回滚并重跑下游。
- 不要求第一版实现完全自由的跨工作流自治执行。

## Agent 能力分层

XiAgent 中的 Agent 应按职责分层，而不是设计为单一的自由执行者。

推荐分层如下：

```text
用户自然语言目标
  ↓
Workflow Supervisor Agent
  ↓
Workflow Runtime / Revision Runtime
  ↓
YAML Workflow
  ↓
Node Agent / Revision Agent / Guard Agent
  ↓
AssetService / ModelService / UserService
```

### Node Agent

`Node Agent` 是工作流节点内部的智能处理能力。

它主要负责处理流程中的开放性任务。

对工作流外部来说，`Node Agent` 仍然是一个普通节点，必须遵守节点输入输出契约。

### Revision Agent

`Revision Agent` 是面向用户自然语言反馈的修订能力。

它主要负责：

- 接收用户对节点结果的修改意见。
- 理解用户希望修改的对象和约束。
- 判断反馈适合局部修订还是可能影响上游。
- 生成修订说明、修订建议或修订后结果。
- 保留原始结果，形成新的修订版本。

`Revision Agent` 默认应优先局部修订，不应默认完整重跑整个节点或上游流程。

### Guard / Critic Agent

`Guard Agent` 或 `Critic Agent` 是质量评审与约束检查能力。

它主要负责：

- 检查节点输出是否满足用户要求。
- 检查角色设定是否一致。
- 检查 JSON 输出是否合理。
- 检查资产引用是否有效。
- 检查分镜是否连贯。
- 检查是否存在安全、权限或数据使用风险。

它可以作为独立节点存在，也可以作为复杂 `Node Agent` 内部的评审步骤。

`Critic Agent` 默认只负责评审和提出修订建议，不直接修改结果、不直接覆盖节点输出，也不负责驱动无限修订循环。

当 `Critic Agent` 输出的是自然语言或高层语义建议时，应由 `Revision Agent` 将其转换为修订计划和修订后结果，再交由 `Revision Runtime` 校验和保存。

当 `Critic Agent` 输出的是结构化、低风险、位于允许范围内的 `patch_candidates` 时，可以由 `Revision Runtime` 校验后直接应用。

### Workflow Supervisor Agent

`Workflow Supervisor Agent` 是工作流级协调能力。

它主要负责：

- 理解用户开放式目标。
- 选择合适的已有工作流。
- 补全工作流输入。
- 串联多个已有工作流。
- 观察任务结果，并在需要时进入修订或追问流程。

它不负责直接修改节点历史记录、资产文件或数据库。

### Skill

`Skill` 是提供给 Agent 使用的场景化能力说明，用于描述特定工作流、领域或任务场景下的规则、标准、操作步骤和输出要求。

`Skill` 不是 Agent 本身，也不是直接执行动作的工具。它主要回答：

```text
在这个场景下，Agent 应该按照什么标准、什么步骤、什么约束来完成任务。
```

以下情况适合引入 `Skill`：

- `Guard / Critic Agent` 在不同工作流中需要不同审查标准。
- `Node Agent` 在不同节点任务中需要不同生成方法。
- `Revision Agent` 在不同修订对象上需要不同修订策略。
- `Workflow Supervisor Agent` 需要根据任务类型选择或串联工作流。
- Agent 涉及特定领域知识、IP 设定、画风规范或业务规范。
- Agent 使用多个工具且工具调用顺序需要遵守固定规程。

第一阶段可以优先支持以下 Skill 类型：

- 审查 Skill。
- 提示词生成 Skill。
- 提示词修订 Skill。
- 分镜生成 Skill。
- 工作流路由 Skill。

普通确定性节点、简单字段转换、简单服务调用不需要 Skill。

## 用户场景

### 场景一：用户要求局部修改提示词

用户看到图像提示词后反馈：

```text
服装更华丽一些，但不要改变发色和姿势。
```

系统应能够理解该反馈主要影响提示词字段，生成修订说明和修订后的提示词，并保留原始提示词版本。

### 场景二：用户指出角色识别错误

用户反馈：

```text
这个角色不是银狼，是卡芙卡。
```

系统应能够判断这可能不是单纯提示词问题，而是上游角色识别或资产匹配问题。第一阶段可以提示需要处理上游结果，后续阶段可支持用户确认后从上游节点重跑。

### 场景三：用户对分镜结果不满意

用户反馈：

```text
分镜节奏太快，中间缺少角色反应镜头。
```

系统应能够识别该反馈影响分镜规划结果，并生成修订建议或新的分镜版本。

### 场景四：节点输出需要质量检查

工作流生成提示词后，系统需要检查：

- 是否包含必要角色特征。
- 是否保留用户要求保留的内容。
- 是否引用了有效资产。
- 是否符合输出 schema。

检查结果应能作为后续继续执行、修订或人工确认的依据。

### 场景五：用户给出开放式目标

用户输入：

```text
帮我把这个剧本整理成可用于分镜生成的角色资产和场景资料。
```

系统应能识别该目标可能需要先运行资产整理工作流，再运行分镜生成相关工作流。第一阶段可以先提供轻量工作流选择能力，后续再支持更完整的跨工作流协调。

### 场景六：不同审查场景加载不同 Skill

同一个 `Guard / Critic Agent` 在不同工作流中需要使用不同审查标准。

例如资产整理工作流关注：

- 角色是否漏抽。
- 变体是否匹配正确。
- 资产引用是否有效。

而分镜生成工作流关注：

- 镜头是否覆盖剧情关键点。
- 镜头节奏是否合理。
- 角色动作是否连续。

系统应支持为 Agent 加载对应场景的 Skill，使同一个 Agent 在不同任务中遵守不同规则。

### 场景七：审查建议进入修订流程

`Critic Agent` 对节点结果进行审查后，可能输出自然语言建议：

```text
提示词缺少服装细节，也没有明确说明保留发色和姿势。
```

系统应支持将该建议交给 `Revision Agent`，由 `Revision Agent` 生成修订计划和修订后结果，再由 `Revision Runtime` 校验并保存新版本。

如果 `Critic Agent` 已经输出结构化修订候选，例如明确的字段路径和替换内容，系统可以在权限、字段范围和 schema 校验通过后，由 `Revision Runtime` 直接应用。

## 功能需求

### FR-1 节点内智能处理

系统应支持将复杂智能任务封装为工作流节点。

节点应具备明确输入、明确输出和输出 schema，并通过节点注册表接入工作流运行器。

### FR-2 节点输出质量评审

系统应支持对关键节点输出进行质量评审。

评审结果应至少能够表达：

- 是否通过。
- 未通过原因。
- 建议修改方向。
- 是否需要人工确认。

### FR-3 用户自然语言反馈接收

系统应支持用户对关键节点结果提交自然语言修改意见。

反馈应与目标任务、目标节点、原始节点输出和修订版本建立关联。

### FR-4 修订触发点

系统应支持在工作流关键阶段设置修订触发点。

修订触发点可以不均匀分布在工作流中，每个触发点应声明其可处理的节点范围和边界。

### FR-5 修订范围约束

系统应支持为修订触发点声明可修订节点范围。

`Revision Agent` 不应越过当前触发点允许的范围修改其他节点结果。

### FR-6 局部修订优先

系统应支持优先对用户明确要求修改的内容进行局部修订。

未被用户要求修改的内容应默认保留。

### FR-7 修订说明

系统应能够生成用户可理解的修订说明。

修订说明应描述：

- 本次修改了什么。
- 为什么这样修改。
- 哪些内容被保留。
- 是否存在上游影响风险。

### FR-8 修订版本保留

系统应保留原始节点结果和修订后结果。

修订不应直接覆盖历史节点输出。

### FR-9 上游影响判断

系统应能够识别用户反馈是否可能影响上游节点。

当反馈涉及角色识别、资产匹配、剧本理解等上游问题时，系统应提示当前问题不适合仅做局部修订。

### FR-10 轻量工作流选择

系统应支持根据用户目标选择已有工作流。

第一阶段可以只支持推荐或启动单个已有工作流，后续再支持跨工作流串联。

### FR-11 场景化 Skill 支持

系统应支持为 Agent 加载场景化 Skill。

Skill 应用于描述特定场景下的规则、标准、操作步骤和输出要求。

系统应支持至少以下使用方式：

- 为审查 Agent 加载不同审查 Skill。
- 为节点 Agent 加载不同任务 Skill。
- 为修订 Agent 加载不同修订 Skill。
- 为 Supervisor Agent 加载工作流路由 Skill。

Skill 不应绕过工作流、服务接口、权限检查或节点输出 schema。

### FR-12 审查建议修订流转

系统应支持 `Critic Agent` 的审查结果进入修订流程。

当审查结果是自然语言建议或高层语义建议时，系统应支持交由 `Revision Agent` 生成修订计划和修订后结果。

当审查结果是结构化修订候选，且位于当前修订触发点允许范围内时，系统可以支持由 `Revision Runtime` 校验后直接应用。

`Critic Agent` 不应直接修改节点输出或持久化修订结果。

## 约束与边界

### 架构边界

Agent 能力必须遵守 XiAgent 的模块化边界：

- Agent 不得直接访问其他模块内部实现。
- Agent 不得直接访问数据库表。
- Agent 不得直接拼接资产文件路径。
- Agent 访问资产必须通过 `AssetService`。
- Agent 触发工作流执行必须通过工作流运行器。
- Agent 相关结果必须保留执行记录或修订记录。

### 工作流边界

当前工作流第一版以 DAG 和条件分支为主，不支持通用循环。

Agent 的多轮生成、评审和修订如果需要循环，应优先限制在单个节点内部，并设置最大轮次。

### 权限边界

Agent 不能扩大用户权限。

所有资产读取、任务读取、任务修订、项目访问都必须经过正式权限检查。

### 持久化边界

Agent 不应直接覆盖已有节点输出。

修订结果应作为新版本、新节点输出或修订记录保存。

### Skill 边界

Skill 只提供规则、标准、步骤和约束，不直接执行系统动作。

Agent 使用 Skill 后，仍必须通过正式工具、工作流运行器和服务接口完成实际操作。

Skill 不应包含可绕过权限、资产服务或持久化边界的实现细节。

### 修订执行边界

`Critic Agent` 负责发现问题并提出建议。

`Revision Agent` 负责把用户反馈或审查建议转换为修订计划和修订后内容。

`Revision Runtime` 负责校验修订范围、校验字段边界、校验输出 schema、生成 diff、保存修订版本和维护审计记录。

在 `Node Agent` 内部的有限自修场景中，实际修改由节点内部生成或修订步骤完成，`Critic Agent` 仅作为评审步骤参与。

## 验收标准

验收标准按 Agent 能力分区拆分。

### P1 验收标准：工作流内部的 Agent 能力

- 可以注册并执行至少一个 `Node Agent` 类型节点。
- 可以对一个关键节点输出进行质量评审。
- 用户可以对指定节点结果提交自然语言修改意见。
- 系统可以生成修订说明。
- 系统可以生成修订后的结果。
- 原始节点结果不会被覆盖。
- 修订结果能够通过目标输出 schema 校验。
- 当用户反馈明显涉及上游问题时，系统能够提示需要上游处理。
- Agent 不会绕过 `AssetService`、用户服务或工作流运行器执行操作。
- 可以为至少一个 `Guard / Critic Agent` 加载场景化 Skill，并按不同审查标准输出评审结果。
- `Critic Agent` 可以输出审查建议，且该建议可以进入 `Revision Agent` 或 `Revision Runtime` 处理流程。

### P2 验收标准：完整观察工作流的辅助聊天与建议 Agent 能力

- 可以读取用户有权限访问的工作流模板、任务事件和节点执行记录。
- 可以解释指定节点的输入来源、输出含义和执行状态。
- 可以根据节点失败、异常或低质量输出生成诊断建议。
- 可以定位与用户问题相关的节点、资产或执行记录。
- 可以提出修订建议、重跑建议或追问问题。
- 可以把建议传递给 `Revision Agent` 或人工确认流程。
- 不会直接修改节点输出、触发重跑或修改资产。
- 不会读取或泄露用户无权限访问的数据。

### P3 验收标准：可以组织工作流的 Agent 能力

- 可以根据用户开放式目标推荐已有工作流。
- 可以说明推荐工作流的理由。
- 可以识别并提示缺失的工作流输入。
- 可以在用户确认或明确授权后启动已有工作流。
- 可以根据任务结果建议进入修订、追问或后续工作流。
- 所有执行动作都通过工作流运行器和正式服务接口完成。
- 不会动态生成并立即执行新的工作流模板。
- 不会绕过 `AssetService`、用户服务或工作流运行器。

### P4 验收标准：跨用户的团队协作 Agent 能力

- 可以汇总多个用户对同一任务、节点、资产或工作流输出的反馈。
- 可以识别重复反馈和冲突反馈。
- 可以生成团队级修订摘要和决策记录。
- 可以根据用户角色、项目权限和任务上下文展示协作信息。
- 可以向相关用户提出澄清问题或协作建议。
- 不会向无权限用户暴露项目、任务、资产或用户反馈数据。
- 不会自动应用存在冲突或高风险的团队反馈。
- 不会替代项目权限、审批和审计机制。

## 分期计划

### P1：工作流内部的 Agent 能力

目标：在不改变当前工作流 DAG 执行模型的前提下，引入工作流内部可控 Agent 能力。

范围：

- 支持关键 `Node Agent`。
- 支持 `Guard / Critic Agent` 对节点结果做评审。
- 支持基础审查 Skill。
- 支持 `Critic Agent` 输出修订建议。
- 支持修订触发点。
- 支持触发点声明可修订节点范围。
- 支持用户对关键节点输出进行局部修订。
- 支持 `Revision Agent` 生成修订计划、修订说明和修订后结果。
- 支持将 `Critic Agent` 的自然语言建议交由 `Revision Agent` 处理。
- 支持保留原始结果和修订结果。
- 支持判断用户反馈是否可能影响上游节点。
- 支持修订历史对比。
- 支持基础修订 Skill。


### P2：完整观察工作流的辅助聊天与建议 Agent 能力

目标：支持一个不直接修改工作流状态的辅助 Agent，完整观察工作流模板、任务事件、节点输入输出快照和修订历史，并向用户提供解释、诊断、建议和追问。

范围：

- 支持查看工作流结构、节点依赖、节点状态和任务事件。
- 支持解释某个节点为什么这样输出。
- 支持诊断失败节点或低质量输出的可能原因。
- 支持基于观察结果提出修订建议、重跑建议或追问问题。
- 支持根据用户问题定位相关节点、相关资产和相关执行记录。
- 支持加载审查 Skill、修订 Skill 和工作流解释 Skill。
- 支持把建议传递给 `Revision Agent` 或人工确认流程。

边界：

- 不直接修改节点输出。
- 不直接触发重跑。
- 不直接改资产。
- 不绕过权限读取其他用户或项目数据。

### P3：可以组织工作流的 Agent 能力

目标：支持 Agent 根据用户开放式目标选择、组织、串联和驱动已有工作流，但仍通过工作流运行器、服务接口和权限检查执行实际动作。

范围：

- 支持工作流推荐。
- 支持工作流路由 Skill。
- 支持补全工作流输入。
- 支持启动已有工作流。
- 支持根据任务结果决定是否进入修订、追问或后续工作流。
- 支持用户确认后的从指定节点重跑。
- 支持低风险结构化 `patch_candidates` 由 `Revision Runtime` 校验后直接应用。
- 支持有限跨工作流协调。
- 支持 Supervisor Agent 与 Revision Agent 协同。

边界：

- 不动态生成并立即执行新的工作流模板。
- 不直接执行节点。
- 不直接访问数据库或资产文件路径。
- 不绕过 `AssetService`、用户服务或工作流运行器。

### P4：跨用户的团队协作 Agent 能力

目标：支持多个用户围绕同一项目、任务、资产和工作流结果进行协作，由 Agent 辅助汇总意见、协调修订、分派待办和维护协作上下文。

范围：

- 支持收集不同用户对节点结果、资产或工作流输出的反馈。
- 支持汇总和去重多用户修订意见。
- 支持识别反馈冲突并向相关用户追问。
- 支持根据用户角色、项目权限和任务上下文分派协作建议。
- 支持生成团队级修订摘要和决策记录。
- 支持更多领域 Skill、团队协作 Skill 和工具使用规程 Skill。

边界：

- 不扩大任何用户权限。
- 不把一个用户或项目的数据暴露给无权限用户。
- 不替代项目权限、审批和审计机制。
- 不自动应用存在冲突或高风险的团队反馈。

## 风险

### 范围膨胀风险

如果把所有用户反馈场景都定义成独立工作流，会导致模板爆炸。

应优先使用少量受控能力和触发点配置，而不是大量场景化工作流。

### Agent 越权风险

如果 Agent 可以直接访问数据库、文件路径或内部服务实现，会破坏权限和审计边界。

必须通过正式服务接口限制 Agent 行为。

### 不可复现风险

如果 Agent 自由决定执行路径且不保存计划、输入和输出，会导致任务不可追踪。

Agent 的关键决策、输入、输出和修订结果必须持久化。

### 用户反馈歧义风险

用户反馈可能非常模糊，例如：

```text
感觉不对。
```

系统不应盲目修改，应支持追问或要求用户明确修改方向。

## 总结

XiAgent 需要的 Agent 能力不是完全自由的自治执行，而是在工作流体系内提供受控智能能力。

第一阶段重点应放在：

```text
Node Agent
+ Guard / Critic Agent
+ Revision Agent 的局部修订能力
```

后续再逐步扩展：

```text
修订触发点
+ 跨节点影响判断
+ 轻量 Workflow Supervisor Agent
+ 受控跨工作流协调
```

这样可以在保持当前工作流可测试、可追踪和可控的基础上，逐步提升系统对开放式用户目标和自然语言反馈的处理能力。
