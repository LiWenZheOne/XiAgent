# 总体平台设计

## 设计目标

XiAgent 第一版要先跑通后端闭环：

```text
用户登录 -> 选择项目 -> 选择工作流模板 -> 创建任务 -> 执行节点 -> 持久化状态 -> 查询任务详情 -> 等待与恢复
```

平台必须为未来前端提供完整底层数据，但第一版不实现前端、不做工作流低代码编排。

## 核心概念

### Workflow Template

工作流模板。由开发者维护的 YAML/JSON 契约定义，支持全局和项目作用域。

### Task

任务。用户选择某个工作流模板并提交输入后产生的一次真实执行。任务是前端任务列表和详情页的主对象。

### Node

节点。模型、工具脚本、人工确认、未来子工作流都必须封装成节点后才能被工作流调用。

### Node Execution

节点执行记录。某个任务中某个节点的一次执行，保存输入快照、输出快照、状态、错误、元数据和 attempt。

### Asset

资产。本地文件或文字资产，可属于全局或项目。节点通过 `AssetService` 获取资产。

### Project

项目。用户的业务空间。项目下拥有任务、项目资产、项目检索结构和项目工作流模板。

## 状态设计原则

工作流状态不是一个被节点反复覆盖的大字典。每个节点执行都有独立记录：

```text
NodeExecution
  ├─ input_snapshot
  ├─ output_snapshot
  ├─ status
  ├─ error
  ├─ metadata
  └─ attempt
```

所谓“当前任务状态”是从节点执行记录和任务事件派生出来的视图。历史数据不覆盖、不删除。

## 接口设计原则

模块外部调用必须走服务接口：

```text
UserService
AssetService
WorkflowService
RuntimeService
NodeRegistry
ExecutionStore
```

模块内部可以迭代，模块外部接口要稳定。跨模块调用不得依赖数据库表结构、文件路径规则或第三方库类型。

## 第三方库边界

LangGraph 是执行引擎适配器，不是平台契约中心。

PydanticAI 是 AI 节点内部实现选项，不是节点接口约束。

FastAPI 是 API 框架，不进入领域层。

SQLite 是第一版存储实现，不进入业务接口。

## 数据持久化要求

第一版需要持久化：

- 用户与项目。
- 工作流模板契约和版本。
- 任务输入、状态、用户、项目、模板版本。
- 节点执行输入快照、输出快照、错误、状态、attempt。
- 任务事件。
- 资产本体、文件存储信息、文字内容。
- 全局和项目级目录树、标签、搜索索引。
- 节点执行中引用过的资产。
