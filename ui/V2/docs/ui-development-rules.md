# XiAgent UI V2 开发子规则

## 适用范围

本文件记录 `ui/V2` 的版本特有 UI 规则。通用规则以 `.codex/skills/xiagent-ui-development/SKILL.md` 为准；本文件只记录 V2 的风格、结构、数据接入、节点控件和验证命令。

## 技术栈

- React 19、TypeScript、Vite、Vitest、Testing Library。最终验收优先使用 Codex 内部浏览器。
- 入口：`ui/V2/src/app/App.tsx`。
- 样式：`ui/V2/src/styles/app.css`。
- API 层：`ui/V2/src/api/`。
- 展示转换工具：`ui/V2/src/utils/display.ts`。

## 产品定位

- V2 是面向最终用户的工作流任务平台，不是静态 demo。
- 核心首屏是项目入口页：一项目一卡片展示项目概况，点击项目后进入该项目的任务工作台。项目页左侧展示项目目录，右侧展示当前焦点项目和创建项目表单。任务工作台只展示当前项目内的任务列表、创建任务、任务详情、等待交互和运行上下文。
- 资产库和项目页是同一工作台的支撑能力，必须和当前项目上下文一致。
- 默认可进入项目来自后端 `/api/projects` 返回的真实 `project_id=global` 全局项目；登录后先展示项目入口页，不直接跳过项目层级进入任务工作台。

## 视觉规则

- V2 采用克制的后台工作台风格：浅灰背景、白色面板、紧凑网格、明确边线、少装饰。
- CSS token 位于 `:root`：
  - 背景：`--bg: #eef2f6`
  - 面板：`--surface: #ffffff`
  - 文字：`--text: #1f2933`
  - 次级文字：`--muted: #65758b`
  - 主色：`--accent: #2563eb`
  - 圆角：`--radius: 8px`
- 主导航为顶部条，任务工作台为三栏布局：左侧项目/任务，中间工作区，右侧运行上下文；窄屏通过媒体查询降为单列。
- 卡片只用于任务、工作流、项目、资产、节点等独立对象；不要把页面大区块层层套卡片。
- 文本必须在按钮、卡片、表单和状态条内完整可读，移动宽度不得互相遮挡。
- 资产卡片必须能容纳超长无空格文件名：卡片不得被文件名撑宽，缩略图必须限制在卡片内部，文件名使用断行而不是横向溢出。

## 数据接入规则

- 登录后调用 `/api/projects`，选择真实项目；默认应为 `global`。
- 用户应先在项目入口页选择项目，再进入该项目工作台；工作台内部不得再维护一个并列的项目下拉来弱化项目与任务层级。
- 任务列表、任务详情、事件流、交互提交、资产查询、工作流查询都必须带当前 `project_id`。
- 创建任务调用 `/api/tasks` 时，body 必须包含当前项目 `project_id` 和选中的工作流契约；业务 `input_data` 不得作为创建任务前置表单要求。
- 工作流必填入参在任务创建后的起始输入节点中填写，提交后由后端固化为 `$workflow.input`。
- 工作流列表调用 `/api/workflows?project_id=<current>`；不能用无项目上下文的全部工作流列表驱动用户创建任务。
- 资产查询使用当前项目与 scope，例如 `scope=combined&project_id=<current>`；资产库范围选择必须提供“项目名资产”“项目名 + 全局”“全局资产”三种模式，按钮文案直接使用当前项目名，并提供清晰可见的选中态。
- 资产库第一版不展示目录结构，也不把目录作为筛选或上传的必要上下文；角色、地点、道具等业务类型优先通过标签快速筛选体现，后续如恢复目录必须重新评估信息架构。
- 上传文件到资产库时必须要求用户填写资产名称，并把该名称作为 `name` 传给 `/api/assets/files`；上传只绑定当前资产范围，分类归属通过标签和业务字段维护。
- 资产库智能上传入口必须在普通上传旁边明确区分：用户提供图片、资产名称、世界背景/原作和资产类型，前端调用 `/api/assets/files/intelligent`，由后端节点补齐 metadata 并自动贴上角色/地点/道具类型标签；前端不得自行拼 LLM prompt 或伪造补全字段。
- 资产详情必须提供现有资产重命名入口，调用真实 `/api/assets/{asset_id}` 更新资产 `name`，不能要求用户重新上传文件来改名。
- 资产库不提供手工填写“文字资产名/文字内容”的创建入口；txt、Word 等文字资料按普通文件上传，后续由资产解析能力处理内容。
- 资产标签过滤和标签新增/删除放在资产库抬头区域；删除标签必须调用真实 `/api/assets/tags/{tag_id}` 接口，且只允许删除未被资产贴过的空标签。
- 资产详情只负责给当前资产贴标签和去标签，必须使用真实 `/api/assets/{asset_id}/tags` 接口；标签选择弹窗必须支持关键词过滤，且只能展示与当前资产 `scope/project_id` 兼容的标签，详情区不得提供新增、重命名或删除标签入口。
- 资产详情里的预览、复制引用、下载、软删除必须使用统一的操作区和一致按钮样式，不要拆成多行风格不同的链接/按钮。
- 资产库页面采用三层工作台结构：顶部保留可搜索项目选择、资产范围和搜索；左侧为上传、类型快速筛选和标签管理；中间为资产列表；右侧为当前资产详情检查器。目录结构在第一版资产库中暂不展示，避免造成无效层级。上传入口不得放在详情检查器底部，避免和当前资产操作混淆。资产详情必须展示缩略图、核心操作、当前标签和可编辑业务字段，业务字段优先包括名称、类型、社会关系、生平背景、当前状态、变体名、变体描述、配件、地点/道具描述；底部再以可读键值网格展示资产记录完整字段，metadata 需要展开为 `metadata.*` 字段，不直接把整段 JSON 原样丢给用户。

## 节点与控件规则

- V2 页面不得直接显示 `input_schema`、`output_snapshot`、`public_url`、节点 ref 或原始 JSON。
- 工作流输入 schema 只能在任务详情的起始输入节点中渲染成可读表单：字符串输入、长文本、选择项、图片资产选择、公开图片地址等。
- 创建任务页不得为 workflow 入参维护独立表单，也不得复制起始输入节点的资产选择、上传或校验逻辑。
- 节点执行输入/输出应渲染为用户卡片：字段列表、文本段落、图片预览、状态、错误和等待操作。
- 节点控件选择遵循工作流优先、节点默认保底：`nodes[].ui` 高于 `workflow.ui.defaults`，`workflow.ui.defaults` 高于 `NodeDescriptor.ui_defaults`，最后才使用系统 fallback。
- 工作流可以分别指定节点的输入、输出、交互和详情控件；工作流只覆盖某一区域时，不应清空其他区域的默认控件。
- 后端 UI 控件 manifest 是控件兼容性的硬性检查来源，V2 React 控件实现必须使用相同 `control_id`、`variant`、`mode` 和 `bindings` 语义。
- 历史任务必须按其持有的 workflow snapshot 和节点 UI 配置展示。禁止在前端把旧 `control_id`、variant、mode 或 binding 静默映射为新控件效果；新样式验收必须通过新建任务生成新 snapshot，或显式迁移/修正目标历史 snapshot/config。
- 可复用节点 UI 已建立在 `ui/V2/src/node-ui/`：
  - `registry.ts` 负责把后端 `control_id` 映射到 V2 React 控件。
  - `resolve.ts` 负责解析任务节点的交互控件配置和 binding。
  - `ControlLibraryPage.tsx` 负责顶部“控件库”页签，并为每个后端 manifest 控件展示真实节点控件渲染出的“节点效果预览”。
  - `controls/ImageViewerControl.tsx` 支持 `ui.display.image_viewer.v1` 的只读图片输出展示，按 `items_path`、`image_url_path`、`label_path` 渲染瀑布流缩略图，并通过点击弹窗查看原图。
  - `controls/ImageChoiceThreeControl.tsx` 支持图片三选一的 `equal_grid`、`hero_list`、`hover_focus` 变体。
  - `controls/SchemaFormControl.tsx` 支持 `ui.input.schema_form.v1`，用于任务详情里的起始输入节点、普通等待输入节点，以及提交成功后的 readonly 参数快照展示。
  - `controls/SchemaFormControl.tsx` 会根据 JSON Schema 字段生成可读标签、placeholder 和 hover 提示；固定枚举字段三个及以内渲染为单选选项组，超过三个渲染为下拉框，不得退回普通文本输入。
  - `ui.input.asset_picker.v1` 是 schema 表单内的单资产选择字段控件，负责按项目范围、资产类型和标签过滤资产库，提交值为选中资产的 `asset_id` 字符串；典型用途是分镜生成工作流首节点按 `集元数据` 标签选择集信息资产。
  - `ui.input.asset_image_picker.v1` 是 schema 表单内的字段级资产图片控件，负责资产库选择、本地上传到资产库后写入结构化图片引用对象；单图字段提交 `{kind:"asset", asset_id:"...", role:"reference"}`，多图字段提交同结构数组，不得把公开 URL 写入工作流输入。资产库页签必须提供项目选择弹窗，项目列表来自 `/api/projects` 并包含 `global` 与用户自建项目，确认项目后目录、标签和资产搜索使用同一项目上下文，项目上下文使用 `scope=combined&project_id=<selected>`，全局上下文使用 `scope=global`。前端可使用公开 URL 进行临时预览，但该 URL 不能进入节点输入、节点输出或 `prompt_results`。
  - `ui.input.script_text.v1` 用于剧本输入节点，提交 payload 仍遵循节点 schema，默认写入 `script` 与 `background` 字段；控件支持直接粘贴文本、拖拽或按钮上传 `.txt`，并在支持 `DecompressionStream` 的浏览器中解析 `.docx` 正文。上传入口必须靠近剧本文本区标题，拖入区域必须有视觉反馈；DOCX 解析失败时必须提示用户另存为 TXT 或直接粘贴，不得阻塞手动输入路径。
  - `ui.interaction.asset_summary_table.v1` 用于 P3 资产列表汇总节点，按角色、地点、道具三个 tab 展示资产列表；一行一个资产，列出该资产已有字段、单一 `匹配资产` 字段和图像列。匹配资产字段可打开按类型过滤且可搜索的资产选择框；列表必须支持新增和删除行，提交 `approved_assets` 时以后端收到的编辑后列表为准；图像列支持逐行选取本地图像上传，提交 `decision`、`approved_assets` 与已补充的 `asset_images`。新增资产的主入口应是 `资产分析`：用户在一个输入框中描述需要新增的资产，前端调用受保护的资产草稿接口，由 LLM 结合用户描述、原始剧本、世界背景和当前资产列表自动判断新增几个资产、分别属于角色/地点/道具哪一类，并补全字段；生成结果只作为预览和修改意见，不得自动写入主表，也不得让主表在“生成前/生成后”出现两套状态。用户点击确认继续时，控件提交当前主表 `approved_assets` 和 `additional_asset_request`，由后续工作流节点解析新增资产。LLM 草稿不得直接入库、不得绕过用户确认，模型不可用时可以提供手动新增当前分类空行兜底。
  - `ui.interaction.asset_image_cards.v1` 用于工作流等待节点中的资产补图工作台，按角色、地点、道具 tab 展示资产卡片；使用该控件的节点必须隐藏通用 input/output/events，只显示业务卡片。卡片标题栏显示“主体_变体_配件”的完整资产名，并把单卡 `生成` / `重新生成` 操作放在标题栏；没有关联资产时显示 `无资产关联`。卡片主体展示图像上传框、关联资产入口、可编辑的主体/变体/配件字段和可编辑提示词，不再展示匹配过程、参考图链接、公开图片 URL 输入或原始输入输出。角色和道具卡片的上传框使用 1:1 正方形，地点卡片使用 16:9 横图；点击上传框选择本地图像，拖拽图片到卡片任意位置也必须触发同一上传逻辑；已有图像时，图像框右下角必须提供悬浮 `下载` 按钮，允许用户单独保存当前卡片图像，且点击下载不得触发文件选择。S17 已匹配到资产库同一资产的条目不得进入 S20 补图和入库列表；S20 的 `关联资产` 只表示生成参考图来源，不等同于 S17 的匹配资产，也不得作为最终 `asset_images` 提交。每张卡片独立提供生成动作：没有图像时显示 `生成`，已有图像时显示 `重新生成`；生成操作只在当前等待节点内调用图像生成 API 并通过 draft 回填图像槽，不推进工作流。顶部 `资产生成` 批量生成当前缺图资产；顶部 `一键入库` 要求所有卡片已有最终图像，将当前 S20 中显示的资产写入资产库，并提交 `decision=finish`、`asset_images`、`prompt_results` 和 `created_asset_ids` 推进到 S22 完成节点。参考图必须使用 `reference_image_ref` 结构化引用，不得使用任何参考图 URL 字段。
  - `ui.display.asset_task_summary.v1` 用于 S22 / P5 任务完成节点，只读展示最终资产概况。控件从节点输出或输入中读取 `asset_images`、`created_asset_ids` 和 `prompt_results`，显示总资产数、角色/地点/道具数量、已入库数量和最终图像列表；提供 `导出为压缩包` 按钮，导出所有最终图像，文件名使用“主体_变体_配件”，同名文件追加序号。
  - 每个控件有稳定 `control_id`。
  - 工作流或节点配置只引用控件 ID、variant、mode、bindings。
  - 控件输入输出必须匹配工作流 schema、节点 descriptor 和运行时交互 payload。
  - 新控件必须补测试，并在本文件记录用途、payload 约束和控件库预览 fixture。
- 任务详情页的节点输入、输出通过 `ui/V2/src/node-ui/` 控件注册表解析；等待交互优先使用控件注册表，未声明交互控件时按节点输出 schema 使用页面级 fallback。
- 节点输入和输出采用上下堆叠布局，不使用左右双栏；默认折叠输入和节点事件，只展开输出或错误区域，避免大输入、大输出节点横向溢出。
- 所有调用 LLM 的节点（节点 ref 以 `ai.` 开头）在输入区必须显示实际提示词信息。通用输入展示控件应在字段列表前提供 `LLM 提示词` 块，展示 `system`、`prompt`、`prompt_template`；当输入包含 `prompt_template` 和 `items` 时，还必须按运行时替换规则预览每个 item 对应的实际提示词，便于调试提取、匹配和生成节点的真实模型输入。
- 任务详情中的节点卡片不显示内部进度条；节点详情只负责展示标题、状态、输入、输出、交互控件和节点事件。
- P 阶段收起态的横向摘要必须使用 V2 工作台风格：浅灰底、细边线、紧凑状态单元和弱状态色；不得使用强装饰时间线、大圆点或与阶段/步骤卡片风格割裂的视觉语言。
- 所有工作流都应优先通过 `workflow.ui.stages` 定义 P 阶段视图；P1/P2 这类顶层单元称为阶段卡片，阶段内部采用 S1/S2 步骤记录流而不是嵌套卡片。任何配置了 stages 的工作流，只有当前页面发起“运行下一步”或“重新运行”后的新增节点才逐步揭示 S 步骤；进入运行态前必须把当前已知事件记为基线，之后只由新推送的节点事件推进步骤显示。切换任务、切换工作流、刷新页面或回看历史任务时，已有真实执行记录必须直接显示，不能播放旧步骤。运行期间不能因为任务详情快照已经包含多个 `node_execution_id` 就一次性预展示本次新产生的阶段节点，也不能出现后续步骤先完成、前序步骤还在运行的视觉错序；已完成历史任务可以直接显示全部真实执行记录。S 步骤的可见性由真实执行/揭示记录决定，但同一 P 内的显示顺序和跨 P 的 S 编号必须按任务持有的 `workflow.ui.stages` 节点声明顺序计算，不能按事件到达顺序、揭示顺序或刷新时机重新排序。P 阶段必须可收起，运行时自动展开包含当前运行、等待、失败或错误步骤的 P；P 标题栏不显示展开/收起按钮，点击标题栏本身展开，再次点击收起，并支持键盘 Enter/Space。P 阶段收起时显示简洁横向线点进度：不使用独立框包裹，圆点代表实际出现的 S 步骤，圆点上方显示 S 编号，圆点下方小字显示步骤名称；P 阶段展开后隐藏该进度条，避免和内部步骤列表重复。S 步骤编号在整个工作流内跨 P 连续递增，不在每个 P 内重新从 S1 开始。每条步骤记录直接显示进度条、百分比、状态和当前动作；运行中的记录不显示独立转圈图标，只在进度条上使用流动动画，等待中的输入节点不显示运行动画。S 步骤详情按是否需要用户处理分类：等待用户输入、确认或选择的步骤自动展开详情；纯运行中、已完成或无需用户动作的步骤默认只显示记录行，用户点击后才展开详情。用户提交当前等待步骤后，前端必须保留该步骤原控件和已填写内容，使用静默刷新等待后端新快照，不能把本地节点临时切成只读输出态导致表单或表格闪空；在下一阶段真实节点事件到达前，可在当前 P 内显示轻量的 `正在准备下一阶段...` 过渡提示，该提示不是 S 步骤、不参与编号。步骤操作必须跟随具体 S 记录显示：当前等待输入步骤显示 `运行下一步`，已完成步骤固定显示 `重新运行`，未到达步骤不显示操作；P 阶段标题栏不显示 `运行下一步` 或 `重新运行`，只保留阶段标题、说明和状态。重新运行前必须提示用户该步骤及下游结果会失效并从此处重跑，历史记录保留，并提供 `修改意见` 输入框；用户填写后前端必须随 rerun API 以 `rerun_revision_note` 发送，后端把它记录在 `node_rerun_started` 事件中，不能把修改意见混入节点业务输入。确认后必须清空该步骤及其下游 S 显示状态，再由新事件重新生成。点击记录后只在该记录下展开节点详情内容，不再出现第二层节点标题卡片。未配置 stages 的旧工作流仅作为兼容路径回退到一节点一卡片视图，新建或重构工作流不得依赖旧回退视图。阶段设计细节见 `2026-05-29-01-workflow-stage-view-design.md`。
- 节点可通过 `nodes[].ui.sections.<input|output|events>.default_open` 覆盖工作流级折叠默认值；也可通过 `visible: false` 隐藏某个区域，通过 `input.wrapper: false` 让输入控件直接显示而不包在“输入”折叠块中。需要用户直接填写输入的起始输入节点通常应直接展示 input，并隐藏无业务价值的 output/events。
- 等待输入节点如果希望把提交动作放到阶段标题栏，应在输入控件配置中声明 `options.submit_placement: stage_header`；控件本体必须使用稳定 form id 接收外部 submit，避免标题栏按钮与控件内按钮重复。
- `ui.interaction.asset_summary_table.v1` 用于资产列表汇总和人工确认。使用该控件的节点必须隐藏通用 input/output/events，只展示业务表格；V2 渲染层也应把该控件作为隐藏通用区块的硬规则，避免旧 snapshot 或漏配工作流把重复的输入输出带回来。表格按角色、地点、道具分 tab；表格主体应采用类似 Excel 的连续网格，不再把每个字段包成小卡片，单元格内容必须自然多行换行，不在单元格内使用额外视觉容器；单元格内的 input、textarea、button 必须继承表格字体，且格子内部不得出现滚动条；名称列、匹配资产列和图像列在所有 tab 中宽度一致。`地点类型`、`时间特征`、`类别`、`关联角色` 使用两字符窄列，`别名`、`配件`、`新增配件` 使用四字符窄列，描述和摘要等长文本字段使用普通宽列。道具默认展示 `描述`、`类别`、`关联角色` 三列，其中关联角色为可选字段。匹配信息只显示一个 `匹配资产` 字段，点击后打开按资产类型过滤并可搜索的资产选择框；图像列使用单一上传格，支持点击选取图像和拖拽上传；新增、删除、修改名称和字段后的结果必须进入提交 payload。
- 顶部导航已有“控件库”页签，用于浏览可用控件、变体、能力标签、绑定要求和预览 fixture；每个 manifest 控件都必须能在该页直观看到节点中的展示样子，字段级控件可通过承载它的节点控件预览展示。控件库页面的控件卡片使用瀑布流列布局，不得因预览高度不同强制行对齐。
- 图片展示类控件可以使用瀑布流列布局，保留图片自然比例和卡片真实高度。资产补图卡片工作台采用 tab 内响应式网格：角色/道具为正方形视觉资产，地点为 16:9 横向视觉资产，不使用瀑布流，也不把上传框压成与内容无关的固定高度。
- `system.user_choice.v1` 在没有工作流显式配置时使用 V2 默认三选一控件；工作流仍可通过 `nodes[].ui.controls.interaction` 改为首图大列表或 hover 放大变体。

## 交互状态

- 必须覆盖 loading、empty、error、disabled、saving、waiting、running、succeeded、failed。
- 等待人工输入时，页面显示问题、输入控件和提交动作；提交内容必须满足人工节点 output schema。
- 创建任务后应自动进入任务详情并选中该任务；切换项目时清理旧项目任务详情，避免跨项目残留。
- 项目页必须显示全局项目，但创建项目表单只创建普通用户项目。

## 验证命令

在 `ui/V2` 下运行：

```powershell
npm run test
npm run build
```

涉及后端契约时，至少运行相关后端测试，例如：

```powershell
python -m pytest tests/test_api_smoke.py tests/test_users_service.py tests/test_runtime_service.py -q
```

最终验收必须用 Codex 内部浏览器和真实后端完成一条主流程：注册/登录、确认当前项目为全局项目、创建任务、进入任务详情、确认页面没有用户不需要理解的 JSON。外部浏览器自动化 CLI、旧 e2e 脚本、直接 API 调用、数据库造数、localStorage 预置或截图检查只能作为辅助回归证据，不能替代内部浏览器真实交互验收。
涉及工作流控件配置变更时，验收主流程必须创建新任务并验证该任务详情使用新的 workflow snapshot；如果使用历史任务验收，必须先执行显式快照配置迁移，不能用旧控件代码兼容来替代配置生效验证。
修改工作流 YAML 后，验收前必须确认运行中的后端工作流目录与磁盘契约一致；工作流模板缓存、运行时 catalog 和任务 snapshot 都不得与当前契约产生未说明的偏差。
