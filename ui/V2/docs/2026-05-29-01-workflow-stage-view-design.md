# V2 工作流阶段视图设计

## 背景

V2 任务详情最初按后端节点一一渲染卡片，适合调试线性流程，但不适合复杂 DAG、条件分支和业务步骤聚合。资产编目工作流需要以 P1/P2/P3/P4/P5 的业务阶段呈现，同时保留后端细粒度节点的输入、输出、事件和错误信息。

## 设计原则

- 后端节点保持细粒度，不为了 UI 合并执行节点。
- 前端顶层显示业务阶段，阶段内部自动生成后端步骤列表。
- 后端 DAG 可以线性、分支、跳过或重试；前端业务阶段顺序保持稳定。
- 用户默认看到阶段状态和已经发生的步骤摘要；需要用户处理的步骤自动展开详情，其余步骤点击记录后才查看节点输入、输出、交互控件和事件。
- P 阶段可以手动收起；任务运行时自动展开包含当前运行、等待、失败或错误步骤的 P；P 收起时显示横向节点进度条，展开后隐藏该进度条。
- 所有新建或重构工作流都应配置阶段；没有配置阶段的历史工作流仅作为兼容路径继续使用旧的一节点一卡片视图。

## 工作流配置

工作流在 `workflow.ui.stages` 中声明阶段。所有面向用户的工作流都应使用该配置表达 P 阶段，除非该工作流仍处在明确的历史兼容或开发调试状态：

```yaml
workflow:
  ui:
    stages:
      - id: p1_input
        name: P1 用户输入
        description: 收集剧本和世界背景。
        nodes:
          - collect_asset_catalog_input
      - id: p2_asset_analysis
        name: P2 资产分析
        nodes:
          - extract_characters
          - lookup_existing_assets
          - match_by_name
```

`nodes` 只声明该阶段包含哪些后端节点，不声明节点内部如何展示。前端会根据 workflow snapshot、node executions 和 events 自动生成步骤状态。新工作流不得依赖“未配置 stages 时一节点一卡片”的回退 UI 作为正式用户体验。

## 阶段状态

阶段状态由内部节点状态聚合：

- 任一节点失败：阶段失败。
- 任一节点等待用户：阶段等待用户。
- 任一节点运行中：阶段运行中。
- 全部节点成功或跳过：阶段成功。
- 全部节点待判定或跳过：阶段待判定。
- 其它情况：阶段待运行。

阶段进度是内部节点摘要进度的平均值。当前进度仍是状态摘要，不是模型或 RunningHub 的真实百分比。

## 步骤记录

P1/P2/P3 这类顶层单元称为阶段卡片。阶段内部不再称为卡片，统一叫步骤记录，并用 S1/S2/S3 编号。S 编号在整个工作流内跨 P 连续递增，不在每个 P 内重新从 S1 开始。步骤记录只来自真实 `node_execution_id`，也就是后端运行时实际执行过或正在等待用户输入的节点；条件分支里没有命中的节点不会提前占位。

任何配置了 stages 的工作流，逐步揭示只用于当前页面主动发起的运行过程，例如用户点击“运行下一步”或“重新运行”之后新产生的节点。进入运行态前，前端必须把当前已知任务事件记录为基线；进入运行态后，只能由基线之后持续推送的新节点事件推进 S 步骤显示。切换任务、切换工作流、刷新页面或回看历史任务时，已有真实 `node_execution_id` 必须直接显示，不能把旧事件重新播放成步骤动画。

当前运行过程中的新增节点应按实际运行顺序进入 S 步骤流。任务详情接口返回的完整 `node_executions` 只作为节点详情数据来源，不能在本次运行过程中绕过揭示队列一次性显示阶段内所有新节点，也不能出现后续步骤先显示完成、前序步骤仍显示运行中的视觉错序。已完成的历史任务可以直接显示全部真实执行记录，便于回看。

步骤记录视觉规则：

- S 编号保持固定显示。
- 每条步骤记录直接显示状态文案、百分比和进度条，避免把进度藏进展开区。
- 运行中的后端节点不使用独立旋转图标；应在进度条本身使用轻量流动动画，并显示当前动作。
- 等待用户的输入节点只显示等待状态，不显示运行动画。
- P 阶段收起时显示横向节点进度条，按真实出现的 S 步骤展示圆点、标题和状态；P 展开后隐藏该进度条，由内部 S 步骤列表承载详细进度。
- 步骤操作跟随具体 S 记录显示：当前等待输入步骤显示 `运行下一步`；已完成步骤固定显示 `重新运行`；未到达步骤不显示按钮，也不占位。
- P 阶段标题栏不显示任何操作按钮，只保留阶段标题、说明和状态；点击标题栏本身展开，再次点击收起，交互方式与 S 步骤记录保持一致。运行与重跑操作必须显示在具体 S 步骤记录上。
- `重新运行` 必须先确认，文案说明该步骤及下游已完成结果会失效并从这里重跑，历史执行记录保留；确认后前端必须清空该步骤及其下游步骤的当前 S 显示状态，再由新的节点事件重新生成步骤记录。
- 完成后保留一条记录；点击记录后在下方展开对应节点详情内容。
- S 步骤详情按是否需要用户处理分类：等待用户输入、确认或选择的步骤自动展开详情；纯运行中、已完成或无需用户动作的步骤默认只显示记录行，用户点击 S 步骤记录后才展开或收起节点详情。
- 阶段视图内不再出现第二层节点标题卡片，避免 P 阶段卡片、S 步骤记录和节点卡片三层标题重复。
- 节点详情沿用统一输入、输出、交互和事件控件，用于调试输入、输出、事件和错误，不参与阶段层级命名。
- 单个节点可以通过 `nodes[].ui.sections` 调整详情区展示：`visible: false` 隐藏 input、output 或 events；`input.wrapper: false` 让输入控件直接显示在步骤详情中，不再包一层“输入”折叠块。
- 资产列表汇总类确认节点应隐藏通用 input、output 和 events 区块，直接显示业务表格控件。表格内只保留一个 `匹配资产` 字段：匹配成功时显示资产名，未匹配时显示 `未匹配到对应资产`；点击该字段弹出资产选择框，选择框必须按当前资产类型过滤，并提供搜索。

## 已接入工作流

资产编目工作流当前按以下阶段显示：

- P1 用户输入：`collect_asset_catalog_input`
- P2 资产分析：角色、地点、道具提取与匹配节点
- P3 用户确认：`review_assets`，审核识别到的角色、地点、道具资产和匹配结果
- P4 资产生成：`filter_assets_for_generation`、`generate_prompt`、`upload_images`、`prepare_asset_images`、`generate_missing_asset_images`。进入本阶段前必须过滤掉 S17 已匹配到资产库同一资产的条目，只处理需要新图像的新资产；`upload_images` 中的关联资产只作为参考图来源。
- P5 最终确定：`merge_uploaded_asset_images` 或 `merge_completed_asset_images`。根据用户选择的 finish / generate_missing 分支只执行其中一个汇总节点，展示最终任务概况，并支持导出所有最终图像压缩包。

资产分镜生成工作流也使用同一套阶段：

- P1 用户输入：`collect_asset_storyboard_input`
- P2 资产分析：角色、地点、道具提取与匹配节点
- P3 用户确认：`review_assets`
- P4 资产生成：`generate_prompt_v2`、`upload_images`、`prepare_asset_images`、`generate_missing_asset_images`、`merge_completed_asset_images`
- P5 最终确定：当前承接后续分镜生成链路，后续如果资产确认和分镜生成拆成两个工作流，可将此阶段收窄为最终资产入库确认

## 后续扩展

- P3 可替换为资产确认表格控件，支持手动关联已有资产或上传参考图。
- S20 资产卡片工作台顶部提供 `一键入库`，把当前 S20 中已有最终图像的可见资产写入资产库；P5 readonly 总结区提供 `导出为压缩包`，文件名使用“主体_变体_配件”。后续如果需要服务端统一治理入库，可再新增 `tool.commit_asset_catalog.v1` 作为后端提交节点。
- 真实进度需要后端运行时发出 `node_progress` 事件，前端再优先读取事件中的百分比。
- P3 资产列表汇总使用 `ui.interaction.asset_summary_table.v1`，必须按角色、地点、道具三个 tab 展示；列表一行一个资产，采用类似 Excel 的连续网格，名称列和匹配资产列保持紧凑，单元格内容自然多行换行，不在格子里再做小框容器，格子内部也不出现滚动条；列出资产字段、单一 `匹配资产` 字段和图像列。匹配资产字段可打开按当前类型过滤且可搜索的资产选择框；列表必须支持新增和删除行，提交时以后端收到的编辑后列表为准；图像列使用单一上传格，支持逐行点击选取和拖拽上传图像，不显示批量公开图片 URL 输入。
